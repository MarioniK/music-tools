# v0.3.0 - genre-classifier LLM migration foundation

## Release metadata

- Release: `v0.3.0`
- Date: 2026-04-30
- Git tag: `v0.3.0`
- Tagged commit: `43cb55b3bc96dd5f9c9ead08d2142ace7a8a0af9`
- Service scope: `genre-classifier`

## Summary

`v0.3.0` closes Roadmap 2 for `genre-classifier` as an LLM migration foundation release.

This release establishes the boundaries, validation rules, evaluation artifacts, and shadow-execution readiness needed for future LLM migration work. It does not move production classification to an LLM provider.

Production behavior remains legacy-safe:

- default provider remains `legacy_musicnn`;
- production response remains legacy-only;
- external `/classify` contract is unchanged;
- response shape is unchanged;
- `tidal-parser` is unchanged.

## Roadmap 2 additions

Roadmap 2 added and documented the following foundation work:

- provider boundary for separating legacy and future provider implementations;
- schema validation for provider outputs;
- compatibility mapping between provider output and the existing response model;
- prompt discipline baseline for LLM-oriented classification work;
- controlled vocabulary baseline for normalized genre output;
- offline comparative evaluation for legacy and LLM-candidate behavior;
- runtime shadow readiness, including configuration and execution design;
- runtime shadow execution baseline;
- controlled live shadow evidence;
- release decision artifact for closing Roadmap 2 and publishing `v0.3.0`.

These items are foundation work only. They do not approve an LLM provider as the production source of truth.

## What did not change

- The default provider did not change and remains `legacy_musicnn`.
- The production `/classify` response continues to come from the legacy path.
- The external `/classify` API contract did not change.
- The response shape did not change.
- `tidal-parser` did not change.
- No canary rollout was performed.
- No LLM production cutover was approved or performed.
- No Python/runtime upgrade was performed.
- No Dockerfile, `docker-compose.yml`, or dependency change is part of this release note.

## Explicit non-goals

The following were explicitly outside the scope of `v0.3.0`:

- LLM production adoption;
- LLM production cutover;
- LLM as the source of truth for `/classify`;
- default-provider switch away from `legacy_musicnn`;
- canary rollout;
- public API changes;
- response-shape changes;
- `tidal-parser` changes;
- Docker or Compose changes;
- Python/runtime upgrade;
- dependency refresh.

## Compatibility notes

`v0.3.0` is compatible with existing `genre-classifier` consumers because the externally visible classification contract remains unchanged.

- Existing callers can continue using `/classify` without request changes.
- Existing response parsers do not need updates.
- Existing `tidal-parser` integration does not need updates.
- Existing deployment expectations remain centered on the legacy MusicNN provider.

Shadow execution and LLM-candidate work are internal migration mechanisms and must not be treated as a changed public contract.

## Operational notes

- Deployments remain legacy-first and legacy-safe by default.
- Shadow execution must be enabled intentionally through runtime configuration.
- Shadow execution must not affect the production response body.
- Shadow evidence should be reviewed as diagnostic migration data, not as proof of production readiness.
- Any future provider switch needs a separate approval and rollout plan.
- Any future canary must be treated as a separate release activity after `v0.3.0`.

## Known limitations

- LLM behavior is not production-approved.
- Live shadow evidence is controlled and limited; it is not a broad production canary.
- The LLM candidate remains constrained by controlled vocabulary, schema validation, and compatibility mapping requirements.
- Runtime modernization remains unresolved in `v0.3.0`.
- Python/runtime upgrade work is deferred to later roadmap items.
- The release does not establish long-term production observability requirements for an LLM provider.

## Next steps after v0.3.0

- Continue post-v0.3 runtime and dependency modernization work separately from Roadmap 2.
- Expand shadow evidence before considering any LLM production rollout.
- Define explicit canary criteria before exposing any LLM output to production consumers.
- Keep `legacy_musicnn` as the default provider until a separate release decision approves otherwise.
- Preserve `/classify` contract compatibility during any future provider migration.
